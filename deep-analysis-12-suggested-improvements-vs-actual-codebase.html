The "bearish bias" claim — is it real?

The claim is that the strategy is skewed toward SELL (bearish) and may collapse in a bull market. This is partially true but overstated. Here is why:

FactImplication
BNKN SELL: 103 trades, 41.7% WR, +4,757 pts (365d)SELL is the dominant edge — true
BNKN BUY: 42 trades, 33% WR, −1,128 ptsBUY underperforms — true
NIFTY BUY: 25 trades, 32% WR, +22 pts (90d) — nearly breakevenNIFTY BUY is not a disaster, just weak
BNKN BUY Apr–Jun 2026 (recent bull months): WR 43–66%, profitableBUY WORKS in uptrends — same strategy, different regime
Aug–Jan period contained multiple RBI-driven down movesHistorical dataset happened to be bearish-heavy, not a flaw in the filters

Conclusion: The strategy is not "broken for bulls". The BUY side has a regime dependency — it underperforms in choppy/bearish markets and outperforms in clean uptrends. The solution is not adding more entry filters (we've proven those block the best trades). The solution is patience — let the market regime rotate.

Top 5 Prioritised Improvements — Honest Audit

SuggestionStatus in CodeReal Assessment
1. ATR-based Stop Loss ✅ Already done Fully implemented. BANKNIFTY uses min(2×ATR(14), 150pts) as SL. NIFTY uses a fixed 45pt cap (ATR range 20–45 pts, so cap is at the high end). The 150pt hard cap was added and saved +2,380 pts. There is nothing more to do here — adding a pure ATR-only SL without the hard cap was tried and produced −368pt losses.
2. Higher Timeframe Confirmation (15m/30m trend) ⚠️ Weekly done; 15m/30m not done Weekly trend filter is already implemented (weekly_trend_buy_ok() using EMA10/EMA20 on weekly candles). A 15m/30m confirmation is a separate idea. Risk: We tried ST age ≥4 candles (proxy for "trend confirmed on higher frame") and it killed the best winning trades. A 15m/30m filter would do the same — the entry fires on a 5m ST flip; by the time the 15m trend confirms, price has moved 50–100 pts already. You'd enter late with a worse RR.
3. Dynamic Intraday Trailing Stop ✅ Already done Fully implemented. The Supertrend trail IS the dynamic intraday trailing stop. It trails the position every candle: BUY → sl = max(sl, st_value); SELL → sl = min(sl, st_value). It only exits when the trend reverses. This is exactly the recommended mechanism — using Supertrend as a trailing stop is already the core of the strategy.
4. Volume + ORB Filter ❌ Not implemented Volume: Angel One returns volume=0 for NIFTY and BANKNIFTY index instruments — they are not traded directly. The VWAP already falls back to equal-weight because of this (see vwap() in indicators.py). A volume filter literally cannot be applied to index instruments via Angel One. Would only work if trading futures/options, not the indices.

ORB (Opening Range Breakout): The infrastructure exists — first_15min_range() is already coded in indicators.py (computes first15_high/first15_low). It is just not wired into the entry signal yet. This could be tested but needs data validation first: does requiring price > first15_high on BUY actually improve WR?
5. Entry Time Optimisation ✅ Already done Fully implemented with data evidence. The report already contains per-slot exit-time analysis. Based on that data:
  • SESSION_START = 09:40 (skip opening 10-min noise)
  • NIFTY_SELL_START = 10:00 (6 fast losers −270pts in 09:40–10:00)
  • BNKN_SKIP_SLOT = 10:00–11:30 (16 trades, 1W/15L, −1,986 pts)
  • NIFTY_BUY_SKIP = 11:30–12:30 (lunch-hour dead zone)
  • NO_NEW_ENTRY_AFTER = 13:00 (13:00–13:30 WR=30% BNKN)
All major dead zones are already filtered. The exit-time table in the report shows exactly where the losses cluster — and those windows are already blocked.

Previous Day High/Low Filter ⚠️ Computed, not wired prev_day_levels() is coded in indicators.py. Not yet used in entry logic. Could be tested: "only BUY if price is above PDH" or "only SELL if price is below PDL." Worth a data analysis run but not a priority — the biggest losses are not near PDH/PDL, they are mid-session whipsaws.
India VIX Filter ❌ Not implemented India VIX is a separate instrument. Angel One provides it (token available) but it requires an additional API fetch per session. Moderate effort. On high-VIX days (VIX > 20), intraday moves are large and SLs get hit. On very low VIX days (<11), moves are small and EOD exits barely reach target. Could filter both extremes. Worth a dedicated analysis run.
Day-of-Week Analysis ⚠️ Thursday skip done for NIFTY Thursday (Nifty expiry) is already skipped. Friday BankNifty is not skipped — it is net positive (+72 pts). Monday and Wednesday have not been analysed separately. Low priority — the report already shows monthly breakdown; day-of-week is a level deeper. The gains from this are likely small.
Professional Metrics (Profit Factor, Expectancy, Sharpe, Sortino) ❌ Not in report Easy to add to the report generator. These are pure calculation additions — no strategy change. Profit Factor = gross wins / gross losses. Expectancy = (WR × avg win) − (LR × avg loss). Sharpe/Sortino need daily P&L series. No risk, purely informational.
Market Regime Analysis (trending/ranging/volatile) ⚠️ ADX filter is a proxy ADX > 20 = trending (required for entry). ADX < 20 = ranging (blocked). ADX > 40 = overextended (blocked). ADX 33–38 dead zone (blocked). This IS a regime filter. A more sophisticated version could use ADX slope or VIX but the current ADX-based approach already captures the core regime distinction.
Opening Range Breakout (ORB) ⚠️ Infrastructure ready As noted above — first15_high/low is already computed. The missing step is wiring it into entry logic and running a backtest to check if "BUY only when close > first15_high" adds value. Our session already starts at 09:40, so the first entry is 25 min after open — ORB would further restrict to only breakout entries. Worth testing in one dedicated run.

Bottom line

3 of the top 5 suggestions are already fully implemented. The other 2 either can't be done (volume) or would likely hurt performance (15m confirmation). The strategy is more complete than the suggestions imply. The real remaining opportunities are: (1) adding better report metrics, (2) testing the ORB filter that already has its infrastructure built, and (3) optionally adding a VIX regime filter. None of these are urgent — the current strategy is profitable across all time windows.
